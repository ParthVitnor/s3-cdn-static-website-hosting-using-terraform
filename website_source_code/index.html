Project Write-up

Static Website Hosting
on AWS with Terraform

We used Terraform to provision an S3 bucket and a CloudFront CDN distribution — no manual console clicking. Here's everything we built and learned.

🪣 AWS S3 ☁️ CloudFront 🔧 Terraform ⚡ Static Hosting
01

What We Built

We deployed a static website on AWS — fully automated using Terraform. Instead of going into the AWS Console and clicking around, we wrote infrastructure as code and ran three commands to get everything live.

The setup has two main parts:

  • S3 Bucket — stores all the website files (HTML, CSS, JS, images)
  • CloudFront Distribution — acts as a CDN, caches your files at edge locations worldwide so the site loads fast for everyone
💡

S3 alone can serve a website, but CloudFront makes it production-ready — faster, more secure (HTTPS), and cheaper at scale.

03

Key Terraform Concepts We Used

Here are the Terraform building blocks that make up this project:

provider

Tells Terraform which cloud to talk to. We used hashicorp/aws with region set to us-east-1.

resource

The main building block. Each resource block creates one real thing in AWS — a bucket, a policy, a distribution.

variable

Lets you pass in values like bucket name without hardcoding them. Makes the config reusable.

output

Prints useful info after apply — like the CloudFront URL so you know where your site is live.

terraform.tfstate

The state file. Terraform uses this to know what's already deployed and only changes what's necessary.

depends_on

Makes one resource wait for another to finish first. We needed this to order the bucket policy correctly.

04

Commands We Ran

These are the exact commands — in order — to get the site live.

1

terraform init

Downloads the AWS provider plugin and sets up the working directory. Run this once before anything else.

2

terraform plan

Shows you what Terraform will create, change, or destroy — before touching anything on AWS. Always check this first.

3

terraform apply

Actually creates the resources on AWS — the S3 bucket, website config, bucket policy, and CloudFront distribution. Type yes when prompted.

4

aws s3 sync

Uploads the website files (HTML, CSS, JS) to the S3 bucket after the infrastructure is ready. The sync command only uploads files that changed.

5

terraform destroy

Tears down everything when you're done. Useful for demos — prevents unnecessary AWS charges by cleaning up all provisioned resources.

05

What We Learned

Honest notes from doing this project — things that clicked, and things that surprised us.

🪣

S3 ≠ a web server

S3 can serve static files, but it's really just object storage. CloudFront is what makes it behave like a proper CDN with caching, HTTPS, and global speed.

📄

State file matters

terraform.tfstate is how Terraform remembers what's deployed. Lose it and Terraform loses track — it may try to recreate everything from scratch.

🔁

IaC = repeatable

The biggest win — destroy the whole setup and recreate it in minutes. No clicking around the console trying to remember what we configured.

🔗

Reference over hardcode

Used aws_s3_bucket.website.arn instead of hardcoding the ARN in the policy. If the bucket name changes, everything updates automatically.

⏱️

CloudFront takes time

Creating the distribution takes 5–15 minutes. That's AWS propagating it to all edge locations globally — not a Terraform issue.

🔒

Public access block

AWS blocks all public S3 access by default. For a public website you have to explicitly disable block_public_policy before the bucket policy works.

06

Common Errors We Hit

Real errors that came up while building this — and how to fix them.

AccessDenied

Bucket policy blocked by public access settings

We applied the bucket policy before disabling the public access block and got an AccessDenied error.

Fix Add depends_on = [aws_s3_bucket_public_access_block.website] to the bucket policy resource.
Invalid Attribute

website {} block inside aws_s3_bucket

We tried putting the website {} block inside aws_s3_bucket. That's the old way — removed in AWS provider v5.

Fix Use aws_s3_bucket_website_configuration as its own separate resource.
403 Forbidden

Site returns 403 after deploy

Bucket policy was applied but still getting 403. The public access block was still partially active.

Fix Make sure all four block_public_* settings in aws_s3_bucket_public_access_block are set to false.
Provider Error

Provider version mismatch

Cached older provider caused resource attribute errors on a different machine.

Fix Run terraform init -upgrade to force-download the correct provider version.
